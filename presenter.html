When the Build System Betrays You

Attackers stopped trying to get into your code. They started targeting the thing that builds your code.

This is a small, self-contained lab. A clean Go web server, a clean git history, a passing security scan — and a backdoored binary at the end of the pipeline. The malicious code is never in the repository. It lives inside a trusted CI builder image and is compiled in silently at build time. It runs as a real GitHub Actions pipeline you can fork and watch, or locally with Docker.

01 Why this matters

It already happened — three times you've heard of

2020
SolarWinds
Malware shipped to 18,000 customers. The source was clean, git was clean, review passed. The implant was injected during the build.
2024
XZ Utils
Two years spent earning trust, then a backdoor slipped into the release tarball — not the source. The git tree looked fine.
2025
tj-actions
One compromised reusable GitHub workflow rippled into roughly 23,000 repositories at once.

Every time, the same move: the attacker never touched the code under review. They poisoned the build — the CI runner, the builder image, the release step — the parts everyone trusts because they have always just worked.

This lab reproduces that pattern end-to-end, on your laptop, so you can see exactly where the trust breaks and exactly which defenses would have caught it.

02 The paradox

Same source. Same commit. SAST passed. Different binary.

The payload registers a hidden, token-gated endpoint that returns 404 to anyone without the secret header — and runs arbitrary commands for anyone who has it:

The full attack chain

  1. Attacker publishes or compromises company/go-builder:latest.A dev laptop with push access, a compromised registry, social-engineered CI access, or a malicious insider. The tag is mutable, so "latest" quietly changes one day.
  2. CI pulls the image by tag — no digest pin, no signature check.
  3. The wrapper copies hook.go into the source directory at build time.
  4. The real compiler builds main.go + the payload together — Go compiles every .go file in the package.
  5. The wrapper deletes the payload — the source tree is pristine again.
  6. The backdoored binary is uploaded as the CI artifact and deployed to production.
What humans saw

PR diff: clean. SAST: passed. Git log: clean. CI logs: go build -o server . Nothing in any review surface ever showed the backdoor.

04 See it run on GitHub Actions

The attack — and the defense — on real CI runners

This isn't a local-only script. Fork the repo, enable Actions, and the supply-chain-demo workflow runs the whole thing on GitHub-hosted runners in three stages:

  1. Unit tests — source is clean.The application's Go tests pass. There is nothing wrong in the repository.
  2. Naive pipeline ships a backdoor.SAST passes, the build succeeds, and it uploads a release artifact — built on GitHub's runner inside the "trusted" image, and backdoored. Download it from the run and strings server | grep __backdoor__ to see it.
  3. Hardened gate catches it.An independent clean reference build plus the reproducible-build gate detect the hash mismatch and block the artifact before it could deploy.
Why this matters

You watch a binary get backdoored by "clean" CI and then watch a cheap, independent check stop it — in the exact environment your own pipelines run in, not on a slide.

Workflow: .github/workflows/supply-chain-demo.yml. The victim's naive pipeline (clean_app/.github/workflows/build.yml) ships a commented, digest-pinned hardened fix alongside it.

05 Reproduce it locally

Prefer the terminal? Run it on your machine

For education and authorized testing only. This repo demonstrates a known attack class against build systems so you can defend against it. Run it on your own machine. Don't deploy the artifacts or point the backdoor at anything you don't own. Safe by default: servers bind to 127.0.0.1, the backdoor's command execution is a dry-run unless you set LAB_ALLOW_RCE=1, and the token is read from the environment (not hardcoded).

Prerequisites: Docker (running) and git — every build happens inside Docker, so no local Go toolchain is needed. Works on macOS and Linux; the scripts auto-detect your OS and CPU. Every command below is also listed in the project README.

06 How to detect & prevent it

Three cheap mechanisms would each have caught this

None of these need new tooling. They need a threat model that includes your build system — and the discipline to enforce it.

Reproducible builds

Build the same source on an independent, known-clean toolchain and compare hashes. Same source, different binary = stop. This is what catches the attack in this lab.

SLSA L3 provenance

Provenance generated by the build platform, not a script — recording the full environment, including the runner image digest. company/go-builder:latest resolves to a different SHA than the expected base; the attestation doesn't match.

Digest pinning

A mutable tag changes silently. Pin to a digest — golang@sha256:… — and the pipeline fails the moment someone swaps the image. That failure is your review moment.

Signature verification

Sign artifacts with keyless Sigstore at upload, verify at deploy with cosign verify-blob. A modified or self-hosted runner won't match the expected OIDC identity.

Scan the builders

Don't only scan apps. trivy image company/go-builder:latest, plus post-build binary SCA with syft + grype on the artifact itself.

Constrain the runner

Ephemeral runners with no persistence between jobs, and a read-only source mount so a wrapper physically cannot drop a file into your package.

This isn't theory in the repo: a real GitHub Actions workflow (.github/workflows/supply-chain-demo.yml) reproduces the attack on every push and then fails the build via the reproducible-build gate — so you can watch the defense catch it in CI, not just on a slide. The victim's naive pipeline ships the included hardened, digest-pinned fix alongside it.

The thesis

Ten years of hardening pushed attackers out of your repo. So they moved upstream — to the runners, builder images, and registry mirrors your developers trust unconditionally. The attack surface moved. Your threat model has to move with it.

Monday-morning questions

  • Who has push access to your builder images?
  • When did you last look inside one of them?
  • Do your artifact hashes match across two independent builds?
  • Are your CI images pinned to a digest, or to latest?

If you can't answer those, you have the same gap this lab just exploited.
